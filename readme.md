## CRYP / HASH 说明

`STM32F407` 的参考手册中包含 `CRYP` 和 `HASH` 外设章节，说明该系列器件在硬件层面具备相关能力。  
但在本项目当前采用的 **官方 `cmsis-device-f4` 精确器件头文件基线** 下，`stm32f407xx.h` 并未提供本项目所需的完整 `CRYP/HASH` 设备级定义链，因此它们暂不纳入 **F407_RT v0.1** 的默认支持范围。

本项目 v0.1 的目标是保持：

- 面向 **STM32F407** 的透明基础库基线
- 仅依赖 **CMSIS + 自写寄存器驱动**
- 不额外引入 **HAL / StdPeriph** 作为默认前提

基于以上原则，当前版本中：

- `CRYP`
- `HASH`

驱动源码可以保留在仓库中，  
但**不作为 F407 主线默认启用模块**，也**不纳入当前最大兼容编译配置**。

后续如需支持这两类外设，可在以下条件满足后再单独扩展：

1. 明确设备头与寄存器定义来源
2. 建立一致的设备支持层
3. 不破坏当前 v0.1 的 CMSIS 基线与可维护性

## USB OTG FS / HS 说明

当前项目所使用的官方 `cmsis-device-f4` 精确器件头文件中，`USB OTG` 的寄存器结构采用全局核心结构与分离类型定义的方式组织。  
在该头文件基线下，当前项目中的 `USB OTG FS / HS` 驱动实现所依赖的 device / host 实例访问模型并未形成完整闭合映射，因此暂不纳入 **F407_RT v0.1** 的默认支持范围。

这并不表示 `STM32F407` 硬件不具备 USB OTG FS / HS 外设，而是表示：

- 当前官方设备头与驱动实现使用的寄存器访问模型不完全一致
- 现阶段仍缺少一层稳定、透明的 USB 设备寄存器访问适配
- 在未完成该适配前，当前驱动尚不能形成闭合能力集

本项目 v0.1 的目标是保持：

- 面向 **STM32F407** 的透明基础库基线
- 仅依赖 **CMSIS + 自写寄存器驱动**
- 不额外引入 **HAL / StdPeriph** 作为默认前提

基于以上原则，当前版本中：

- `USB OTG FS`
- `USB OTG HS`

驱动源码可以保留在仓库中，  
但**不作为 F407 主线默认启用模块**，也**不纳入当前最大兼容编译配置**。

后续如需支持这两类外设，可在以下条件满足后再单独扩展：

1. 明确当前官方设备头下的 USB device / host / endpoint / channel 访问模型
2. 建立一致的 USB 寄存器访问适配层
3. 不破坏当前 v0.1 的 CMSIS 基线与可维护性
